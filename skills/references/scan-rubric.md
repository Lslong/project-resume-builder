# 扫描与推断准则(scan-rubric)

阶段 1 用本文件指导：扫什么、怎么判岗位、怎么挑功能点、怎么守真实性。 全程只读项目代码，不修改项目任何文件。

## 一、扫描维度

派分析 agent 收集：

1. 技术栈：语言、框架、关键库、构建工具、状态管理、后端/存储。来源：package.json / requirements.txt / go.mod / pom.xml 等依赖文件、配置文件、import 语句。
2. 整体架构：目录组织、分层方式、模块边界、数据流向。
3. 各模块职责：每个主要目录/模块做什么。
4. 关键功能点：见"三、关键功能点筛选(深度优先)"。
5. git 信息(若有)：提交规模、活跃模块——仅作参考，不用于判定模块归属(本 skill 不区分谁负责)。

## 二、岗位方向：简单启发式

不写复杂规则，按技术栈一眼判断：

+ 出现 React / Vue / Angular / 浏览器 API / CSS 工程化 / Webpack / Vite / 小程序 等 → 前端
+ 出现 PyTorch / TensorFlow / transformers / LangChain / LLM API / 向量库 / 推理优化 / RAG 等 → AI 开发
+ 两者都重 → 取占比更大的一方，或在产物里注明"全栈偏 X"
+ 用户传了 --role → 以用户为准，忽略自动推断

岗位方向只影响 bullet 措辞侧重，不影响诚信规则：

+ 前端侧重：性能优化、工程化、交互体验、框架原理、浏览器机制、状态管理
+ AI 侧重：模型/方案选型、prompt 工程、推理与成本优化、数据链路、评测

## 三、关键功能点筛选(深度优先)

不要罗列所有功能。真实简历的杀手锏是 1-2 个能深挖 20 分钟的亮点，而非 8 条平庸 bullet。"什么都做过、什么都不深"恰恰是初级候选人最容易被刷的画像。

筛选标准(满足越多越该写)：

+ 有真实技术含量(不是 CRUD / 调库的默认用法)
+ 能讲清"为什么这么做"
+ 体现工程权衡、性能优化、复杂状态管理、架构设计、算法/数据结构、疑难问题解决

操作步骤：

1. 先全局扫一遍，列出候选功能点。
2. 挑出 1-2 个最有深度的作为重点，深入推演(方案对比 + 选型理由 + 难点)。
3. 其余有价值的点压缩成简短条目；无技术含量的直接丢弃。

注意：简历(阶段 2)只取精选；面试稿(阶段 3)覆盖全部有价值技术点。筛选结果两边复用，详略不同。

## 四、方案推演 + 选型理由

对重点功能点，推演 2-3 种可选技术方案，论证为什么现在这个最优。这是真实可辩护的工程判断。

必须套可讲性止损(见第六节)：推演要基于代码现状站得住，不能写成候选人答不出的教科书式对比。凡是从代码里看不出"确实做过这个权衡"的，降级为事实陈述或标注建议删除。

## 五、亮点设计点(独立于方案推演)

有些精巧设计本身就是实力证明，但套不进"为什么不用方案 B"的框架，例如：

+ 一个优雅的抽象 / 接口设计
+ 一个巧妙的数据结构或缓存结构
+ 一个关键的性能手法
+ 一个清晰解决复杂问题的思路

这些一定要单独收集，进面试稿的"亮点设计专区"。不要因为套不进推演框架就漏掉能体现技术实力的设计。

## 六、四层真实性判定(硬规则，不得违反)

+ **第一层 技术选型论证 / 架构理解**：代码可证，写实。
+ **第二层 量化指标**：仅性能/规模/效率类留【XX%】占位 + 测量建议；选型/架构类不塞占位符，靠技术深度；绝不编造任何数字。
+ **第三层 难点**：只写代码里真实可见的复杂度，不伪装成"我当时试了 B 然后失败"这类亲历经历。
+ **第四层 可讲性止损**：每条论证须能口头复述一句话内核；讲不出真实权衡的 → 降级为事实陈述，或标"讲不出建议删除"。

第四层是把"不编造"升级为"抗追问"的关键：只写候选人 hold 得住的内容。

## 七、零源码 / 伪代码边界

+ 产物不贴项目真实源代码(隐私 + 通用性)。
+ 简历素材里完全不出现伪代码或代码。
+ 面试稿里：仅"可能被面试官要求白板手写"的点，可用伪代码 + 自然语言辅助讲清逻辑，且需明确标注是伪代码。
+ 始终优先用自然语言描述实现思路，而非贴代码。

## 八、模块归属(不区分)

不去判定"谁写的"，所有有价值的功能点都默认按用户主导产出，采纳权交用户。仅在简历末尾附一行自查提示，点出明显是框架/库自带能力、建议用户自行核实的条目。
